Integrating Pricoin deposits and withdrawals

Pricoin's RPC surface follows Bitcoin Core conventions, with one important addition: every non-coinbase output is confidential, and deposit attribution uses subaddresses instead of per-user transparent addresses. This page maps the standard exchange-integration verbs to the actual RPCs and walks through both the deposit and withdrawal paths end-to-end.

If you've integrated Bitcoin before: the daemon starts the same way (pricoind, JSON-RPC on port 8332/18443), the wallet model is the same, and most of the familiar verbs (getblock, getrawtransaction, getblockchaininfo) work unchanged. The differences live in three places: how you allocate deposit addresses, how you detect incoming credits, and how you broadcast withdrawals.

Status: this guide tracks Pricoin v0.1.x. Treat it as a working draft — RPC names and JSON shapes can still change. Open an issue on GitHub if anything looks wrong or incomplete.

Methods cheat-sheet

The conventional exchange-integration verbs and what they map to in Pricoin's RPC surface. Anything that doesn't appear here behaves identically to upstream Bitcoin Core.

Conventional verbPricoin RPC(s)Notes
getNetworkInfo getnetworkinfo
getblockchaininfo		 Same as Bitcoin. Use getblockchaininfo.blocks for confirmation tracking.
getNewAddress pricoin_getnewsubaddress [label] Allocates a fresh deposit subaddress (index ≥ 1). One per user. Returns pricsub…-prefixed encoding that carries the index in its payload.
validateAddress validateaddress <addr> Recognises transparent (pric…), stealth-main (pricstl…), and stealth-subaddress (pricsub…). The kind field tells you which.
getBlock getblock <hash> [verbosity] Same as Bitcoin. CT outputs appear with their commitments and rangeproofs in the verbose form.
getTransaction gettransaction <txid>
getrawtransaction <txid> 1		 Returns header metadata and the raw v4 transaction. Amounts are blinded — use pricoin_listownct to discover values payable to your wallet.
getRecentTransactions pricoin_listownct The deposit-discovery RPC. Returns one row per recovered output with txid, vout, value, height, subaddress_index, and subaddress.
sendTransaction walletsendct <dest> <amount> <fee>
walletsendct_ring (with ring sig)		 Withdrawal path. dest can be transparent, master stealth, or subaddress — the wallet branches on kind automatically.

The legacy sendtoaddress/sendmany/send/sendall calls return RPC_METHOD_DEPRECATED on Pricoin — consensus rejects v1/v2/v3 transparent transactions. Withdrawals always go through the CT send path.

Deposits — attributing incoming funds to users

Pricoin's stealth model means every output is private — even with the wallet's view key, an outside observer cannot link two payments to the same identity. To still tell deposits apart by user, the exchange hands each user their own subaddress: a deterministic per-index identity derived from the wallet's master seed. All subaddresses share one view scalar, so a single wallet recovers payments to any of them.

  1. Allocate one subaddress per user

    When a new user registers, call pricoin_getnewsubaddress "<your-internal-user-id>". The returned address is what you display on the deposit screen; the index is what your DB pins the user to. 

    $ pricoin-cli -rpcwallet=hot pricoin_getnewsubaddress "user-4815"
{
  "index": 47,
  "address": "2qdRXWpkrmNVLad...AqqCKWe1RnHxFNgV5L7JSD8",
  "view_pubkey":  "030f99...ba536514",
  "spend_pubkey": "02fac0...c231ffdec",
  "label": "user-4815"
}

    Indices are monotonic and never reused. Backups are still just the wallet seed — the seed deterministically reproduces every subaddress for restored wallets.

  2. Poll pricoin_listownct for new credits

    On a schedule (every block tick, or every N seconds — your call), run: 

    $ pricoin-cli -rpcwallet=hot pricoin_listownct
{
  "scanned_blocks": 0,                  ← 0 means warm cache hit
  "scanned_mempool_txs": 0,
  "total_recovered": "32.50000000",
  "outputs": [
    {
      "txid":             "ab12...",
      "vout":             1,
      "value":            "5.00000000",
      "height":           812347,
      "subaddress_index": 47,
      "subaddress":       "2qdRXWpkrm...JSD8"
    },
    ...
  ]
}

    The first call after a fresh daemon start pays for a full chain scan; subsequent calls only walk the new tip-extension. Steady-state polling is cheap.

  3. Attribute by subaddress_index

    Map each row's subaddress_index to the user you allocated it to. 0 means the master stealth address (handle as you see fit — typically internal/sweep flows, not user deposits). The encoded subaddress field is also surfaced so you can key directly off the string if your DB stored that instead.

  4. Track confirmations

    Compare each row's height against the current tip from getblockchaininfo.blocks. Apply your confirmation policy (Pricoin's block time is 150 s, so 6 confirmations ≈ 15 minutes). Mempool entries report height = -1.

  5. Don't double-credit on rescans

    pricoin_listownct returns every currently-unspent recovery — outputs whose key images haven't yet appeared on chain. Once you sweep a deposit (i.e. include it as an input in a withdrawal), its key image lands on chain and the row disappears from subsequent listings. Until that point, the row repeats on every poll: track which (txid, vout) pairs you've already credited.

One wallet, many users. A single hot wallet hands out subaddresses for every user. There is no per-user wallet sprawl, no loadwallet overhead, and no per-user rescan. The subaddress lookup table is built once per scan pass and amortised across all blocks walked.

Withdrawals — sending PRIC to user-supplied addresses

Users withdraw to addresses they control: a transparent pric…, a stealth master pricstl…, or a subaddress pricsub…. Validate the input first, then hand it to the CT send path.

  1. Validate the destination

    Call validateaddress. The kind field is what to branch on: 

    $ pricoin-cli validateaddress "2qdRXWpkrmNVL...JSD8"
{
  "isvalid":          true,
  "kind":             "stealth_subaddress",
  "address":          "2qdRXWpkrmNVL...JSD8",
  "view_pubkey":      "030f99...",
  "spend_pubkey":     "02fac0...",
  "subaddress_index": 47
}

    Reject withdrawals where isvalid is false. Other kinds (transparent, stealth_main, stealth_subaddress) are all valid destinations — the wallet's send path handles them correctly without per-kind logic on your side.

  2. Broadcast via walletsendct

    Pricoin's CT-aware send RPC. Takes the destination, PRIC amount, and miner fee: 

    $ pricoin-cli -rpcwallet=hot walletsendct \
    "2qdRXWpkrmNVL...JSD8" 1.5 0.0001
{
  "txid":              "...",
  "ct_outputs":        3,
  "dest_was_stealth":  true
}

    For higher-anonymity withdrawals use walletsendct_ring <dest> <amount> <fee> <ring_size> which signs the input with a CLSAG ring of the given size (default 4; 16 is the conservative recommendation for user funds in a long-running hot wallet).

  3. Watch the txid

    Track confirmations the same way you do today — gettransaction <txid> returns the header info (status, confirmations, blockhash). The on-chain tx is a v4 confidential transaction; the wallet has all the metadata it needs to display it back to operators.

The wallet branches on address kind automatically. Subaddress payments compute R = r·Bi instead of R = r·G, which the wallet handles internally — you don't need to know about this distinction. Indistinguishability on chain is preserved either way.

Confirmations & reorgs

Standard Bitcoin-style depth-based confirmation policy applies. A few specifics worth knowing:

  • Block time is 150 s, vs Bitcoin's 600 s. 6 confirmations is ~15 minutes; pick a depth that matches your risk model. For exchange-grade attestation, 12–20 confirmations is conservative.
  • The recovery cache (the per-wallet structure pricoin_listownct uses to avoid re-scanning) detects reorgs automatically. If the chain at the cached last_scanned_height no longer hashes to the remembered block hash, the cache is dropped and a full fresh scan runs on the next call. You don't need to do anything.
  • Outputs disappear from pricoin_listownct when their key image is seen on chain — i.e. when something (your wallet or a successor of it) has spent them. Never re-credit a deposit on disappearance — that means the funds moved out, not that they were rolled back.
  • getblockchaininfo reports initialblockdownload: true until headers and blocks are caught up. Hold off on credit-decisions until that flips to false after a fresh daemon start.

Operational notes

Wallet must be unlocked for scanning

pricoin_listownct needs the master view scalar to derive shared secrets, and the master spend scalar to derive one-time spend keys. Both live in the wallet's encrypted stealth record. If the wallet is encrypted, call walletpassphrase <passphrase> <timeout_sec> first. A common operations setup: encrypt the wallet at rest, unlock it on daemon start via a wrapped secret in your secrets manager, re-unlock periodically before the timeout expires.

Backups

The wallet's stealth seed reproduces every subaddress deterministically. pricoin_getstealthseed returns the 32-byte seed (when the wallet is unlocked) — that's the single value you back up. The standard backupwallet RPC also captures everything: subaddress indices, labels, swap records, transparent descriptors.

Transaction sizes

Per-output Borromean rangeproofs are ~1.3 KB; per-input CLSAG ring sigs ~600 B. A typical 1-in/2-out withdrawal serialises to ~3.5 KB — about 2× a comparable Bitcoin tx. Block weight limits are raised to accommodate this; mempool admission and fee estimation work the same way as upstream.

Deposit minimum

Set a deposit minimum that comfortably covers your own per-output handling cost (sweeping aggregated deposits to cold storage costs ~3.5 KB per send). Dust-value deposits cost more to consolidate than they're worth.

Hot/cold split

Standard pattern: hot wallet runs pricoind, holds the master stealth identity, and serves pricoin_listownct polls + walletsendct withdrawals. Cold wallet holds a separate identity, sweeps from hot to cold via walletsendct on a schedule. Cold-wallet pricoind can be air-gapped; it just needs the chain blocks (rsync from hot, or a separate full node) and its own seed-derived view+spend keys.

Atomic swaps (optional)

If you want to expose BTC↔PRIC or LTC↔PRIC swaps directly to users without listing PRIC on-exchange, the wallet ships a Nostr-relayed orderbook + adaptor-signature swap protocol. See the whitepaper §11–12 and the homepage swap section.

Reference snippets

Common one-liners for ops dashboards / cron jobs.

Sync watch

# Block until the daemon is fully synced (initialblockdownload=false).
while true; do
  ibd=$(pricoin-cli getblockchaininfo | jq -r .initialblockdownload)
  [ "$ibd" = "false" ] && break
  sleep 5
done

Sweep a user's deposit credits

# All unspent CT outputs paying user N (subaddress_index == N), summed.
pricoin-cli -rpcwallet=hot pricoin_listownct \
  | jq --argjson N $N '[ .outputs[] | select(.subaddress_index == $N) | .value | tonumber ] | add // 0'

Address validation in your withdrawal endpoint

# Returns 0 (valid) or 1 (invalid). Surface the JSON to the user.
addr=$1
res=$(pricoin-cli validateaddress "$addr")
echo "$res" | jq -e '.isvalid' >/dev/null && exit 0 || exit 1

List all subaddresses with their labels

pricoin-cli -rpcwallet=hot pricoin_listsubaddresses \
  | jq -r '.[] | "\(.index)\t\(.label)\t\(.address)"'
